# 契约模式

关于如何写出"一次过审"的 Goal Contract 的实战笔记。

本文的每一条都来自一次真实的构建:两天内通过这座桥交付了七份 Goal Contract(正是本仓库自己的 Phase 2–4 功能——持久化、重试、并发、依赖链、停滞看门狗、成本预估)。七份合同全部一次通过审查,没有发出过一份 Delta Contract。下面就是让这件事发生的模式。

*[English version](./PATTERNS.md)*

## 一份有效合同的解剖

本仓库的真实合同(Task 3.3,任务依赖链),带注解:

```markdown
### Goal
codex_implement 必须支持声明对先前 job 的依赖:被依赖的 job 成功后,
依赖它的 job 自动启动;失败则依赖方快速失败并附明确原因。

1. Schema:codex_implement 新增可选参数 `depends_on: string` ……
2. 提交时校验:未知 job -> isError,不创建 job;……
3. 阻塞语义:…… blocked 状态的 job 不占用并发槽位 ……
4. 计时与事件:总超时从 job 实际启动时才开始计 ……
5. 持久化:…… 重启加载时 "blocked" 状态标记为 interrupted ……
```

它有效的原因:

- **第一句话就是验收测试的散文版。** 不是"改进任务编排"(愿望),而是"依赖成功后自动启动、失败则快速失败"(可以核对的行为)。
- **编号条目预先解决所有需要判断的问题。** blocked 的 job 占不占并发槽位?超时从什么时候开始计?重启后怎么办?每一个都是实现者原本会默默替你做的决定——一次你只能在审查时才能抓到(或漏掉)的掷硬币。
- **边界语义写明,而非暗示。** "未知 job → isError,*不创建 job*"同时锁定了错误行为和副作用边界。

目标不是把合同写长,而是**留下零个你会在审查时打回的开放设计决策**。凡是你会在审查里挑的刺,就在合同里先定下来。

## Constraints:圈定爆炸半径

```markdown
- 只修改 mcp/src/jobs.ts、mcp/src/server.ts,及 mcp/test/ 下的新测试文件。
- Do not modify:contracts.ts、codex-client.ts、log.ts、store.ts、config.ts、docs/、README*。
- 不新增运行时依赖。工具 schema 只能增量扩展。
- 持久化格式必须向后兼容:没有新字段的旧文件必须能正常加载。
- 现有 N 个测试保持通过;只允许最小必要调整。
```

值得保留的模式:

- **显式的允许清单 + 显式的禁止清单。** 允许清单圈定你将要审查的 diff 范围;禁止清单保护承重文件。审查的第一步就是花 5 秒把 `git status` 对照这两张清单。
- **对一切公共表面(schema、持久化格式、API)要求"只许增量扩展"。** 这一行防住所有静默的破坏性变更。
- **写明当前测试的确切数量**("现有 29 个测试保持通过")。它让"别弄坏已有的东西"变得可核验,数字的漂移本身就是审查信号。
- **预期有原则的越界。** 在本仓库的实战中,Codex 有一次修改了两个严格来说在允许清单之外的旧 mock 文件——因为一个新默认值(重试开启)改变了旧 mock 的语义。那是*正确的*工程判断。裁量越界时看它是否被合同的 Success Conditions 逼出来;好的 handoff 会主动说明理由。

## Success Conditions:让机器来当裁判

最重要的一条规则:**至少有一个条件是由退出码说了算的命令。** 其他一切都建立在这之上。

以下验证技巧全部出自本仓库的测试套件,并且都在关键时刻起了作用:

| 要证明…… | 就这样做 |
|---|---|
| "拒绝时没有创建任何 job/记录" | 调用前后数状态目录里的文件,断言相等。 |
| "实际送达的 prompt 里包含 X" | 让 mock 在缺少 X 时*拒绝请求*(`EXPECT_*` 环境变量)。job 能跑成功本身就是证明。 |
| "从未 spawn 过进程" | 用两条独立通道断言:结构化日志里无 spawn 事件,*且* transcript 里无 attempt/thread 记录。 |
| "文件内容从未被内联进 prompt" | 往文件里写一个标记串,断言 prompt 里没有它。 |
| "B 只在 A 结束后才启动" | 断言 progress 事件流的*顺序*,而不只是最终状态。 |
| "超时从出队开始计,而不是入队" | 让总耗时超过配置的超时并带排队;能跑完本身就证明了计时起点。 |
| "时序敏感的测试不 flaky" | 接受之前先用 `bun test --rerun-each 3`(或等价手段)压一遍。 |

还有关于 handoff 的元规则:**把它写进合同。** 这里的每份合同都以这一条收尾:

```markdown
- [ ] Final handoff 列出:Changed Files、Validation(命令 + 输出)、
      每条 Success Condition 的核验结果、Risks & Deviations。
```

缺失或不完整的 handoff 本身就是审查失败——你永远不需要为此争论,因为它就是一条成功条件。

## 审查:永不轻信,永远重跑

handoff 说 `bun test` 过了。你还是要自己跑一遍 `bun test`。在本仓库的实战中,每一份 handoff 碰巧都是诚实的——而这恰恰是你不能假设的事,因为哪天有一份不诚实,你的重跑就是唯一的防线。行之有效的审查循环:

1. `git status` 对照允许/禁止清单(5 秒)。
2. 读 diff——不止看对错,还看设计。(这里最好的两个发现都是*正面的*:一次公平性修复和一次停滞边界的精化,都是合同没要求的。)
3. 亲自从干净的 shell 重跑每一条验证命令。
4. 把 handoff 的 Risks & Deviations 和 diff 的实际内容对照。"None" 加上一个未声明的偏差,比声明了偏差更危险。主动写出"P90 用的是 nearest-rank 方法"的 handoff 是好信号。

## Delta Contract:只圈定失败项

审查失败时,不要重新谈判整个任务。只发回:

```markdown
### Findings
- store.ts:41 — interrupted 的 job 保留了过期的 endedAt;重启测试失败。

### Failed Success Conditions
- [ ] 持久化往返测试通过。

### Constraints
- 原有约束继续适用。只修复上述发现;不要碰已通过的部分。
```

线程(连同它的 goal)会被恢复——Codex 保有它自己实现的完整上下文,所以修复是外科手术式的。本仓库的七个任务零 Delta Contract;诚实的归因是前期合同写得足够精确,而这比任何返工循环都便宜。

## 校准数字(一个本地数据点)

来自构建 Phase 2–4 的七个 implement job(TypeScript,单个 diff 约 40–440 行,`reasoning_effort: high`):

- **单任务 token:** 51k–128k(中位数 ≈ 87k)。`codex_estimate` 现在能给出你自己机器上的数字——优先用那个。
- **单任务耗时:** 健康时 5–15 分钟;有两个 job 在 turn 中途停滞了 10–16 分钟(上游 CLI 问题)后自行恢复。
- **超时:** 30 分钟的默认值兜住了两次停滞。如果你要收紧它,先看停滞看门狗的 `seconds_since_activity` 再判断 job 是否死了——从外面看,深度推理的暂停和真正的停滞长得一模一样。
- **Effort:** `high` 产出了超出合同要求的判断(公平性修复、边界精化),默认档没有。对边界语义重要的合同,它物有所值。

## 反模式

- **愿景式目标。** "把 server 做到 production-ready。" Codex 会自己决定这是什么意思;你会在审查时不同意。
- **不可机验的条件。** "代码整洁、测试充分。" 机器判不了的,就是审查备注,不是成功条件。
- **无围栏的 diff。** 没有文件允许清单 → diff 蔓延 → 审查成本爆炸 → 细微回归藏进噪音。
- **隐式边界。** 你留下的每一个"如果 X 会怎样",都会被实现者默默决定——并且是对着你的审查清单反着来。
- **轻信 handoff。** 不重跑就接受"所有测试通过",那不是审查,是多了几个步骤的祈祷。
- **巨型合同。** 一份合同对应一个可审查的增量。七份小合同加干净的 handoff,永远胜过一份史诗级合同——而且真出问题时,返工循环还能保持外科手术式。
